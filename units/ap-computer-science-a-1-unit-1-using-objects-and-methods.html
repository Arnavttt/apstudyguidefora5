Primitive Types and Variables

Learn on this site
Primitive Data Types

Java has eight primitive types; AP® CS A focuses on int (whole numbers), double (decimals), and boolean (true/false). Literals are written directly: 42, 3.14, true.

Common misconception: String is NOT a primitive — it is a class/reference type. Confusing String with primitive types is a common error.

How it can be tested: MCQ identifies type of a value; FRQ declares correct types in method signatures.

Variable Declaration and Assignment

Declare: type name; or type name = value; Assignment uses = (not equality). Java is statically typed — the type cannot change after declaration.

Common misconception: The = operator is assignment, not equality. == tests equality for primitives.

How it can be tested: Trace code to find the value of a variable after a series of assignments.

Casting and Arithmetic

Integer division truncates: 7/2 → 3. Cast with (double) to get 3.5. Modulo % gives remainder: 7%2 → 1. Mixed int/double arithmetic promotes to double.

Common misconception: Integer division silently truncates — a frequent source of off-by-one errors in AP® FRQ.

How it can be tested: Predict result of arithmetic expressions including integer division and modulo.

Key Vocabulary
  • int — Primitive integer type; stores whole numbers; range ±2.1 billion
  • double — Primitive floating-point type; stores decimals; IEEE 754
  • boolean — Primitive type; stores true or false
  • assignment — The = operator stores a value into a variable; does not test equality
  • modulo % — Returns remainder of integer division; e.g., 7%3 = 1
Objectives
  • Declare and initialize int, double, and boolean variables.
  • Trace arithmetic expressions involving integer division and modulo.
  • Explain the difference between primitives and reference types.
How this can be tested
  • MCQ: evaluate expression with integer division and casting.
  • FRQ: declare a variable with the correct type in a method.
  • Tracing: find the final value of a variable after several assignments.
Practice before moving on
  • Practice arithmetic operator precedence and integer division.
  • Write declarations for a small program tracking a score and a balance.

Objects and Method Calls

Learn on this site
Classes and Objects

A class is a blueprint; an object is an instance. Create objects with new: String s = new String("hi"); or String s = "hi"; Objects live on the heap.

Common misconception: Using a variable name before assigning an object gives a NullPointerException, not a compile error.

How it can be tested: Identify whether a variable holds a primitive or a reference; trace object creation.

Calling Instance Methods

Instance methods are called on an object: object.method(args). They may return a value or be void. The dot operator accesses members of an object.

Common misconception: Methods called on null throw NullPointerException — check that the variable has been initialized first.

How it can be tested: Read and trace method calls; predict return values from method documentation.

String Methods

Key String methods (AP® CS A subset): length(), substring(i,j), indexOf(str), equals(str), compareTo(str). Strings are immutable — methods return new Strings without changing the original.

Common misconception: Strings are compared with .equals() not ==. Using == compares references, not content.

How it can be tested: Apply String methods in code and predict return values; trace substring indices.

Key Vocabulary
  • class — Blueprint that defines data (fields) and behavior (methods) for objects
  • object — Instance of a class created with new; holds its own copy of instance variables
  • null — Default value of a reference variable before initialization; accessing it causes NullPointerException
  • String.substring(i,j) — Returns characters from index i (inclusive) to j (exclusive)
  • String.equals(s) — Compares String content; use instead of == for Strings
Objectives
  • Create objects using the new operator.
  • Call instance methods and interpret return values.
  • Apply String methods: length, substring, indexOf, equals.
How this can be tested
  • MCQ: identify the output of a sequence of method calls.
  • FRQ: call the correct String methods to manipulate a String parameter.
  • Trace NullPointerException scenarios.
Practice before moving on
  • Practice all String methods from the AP® CS A Quick Reference.
  • Write code to extract a substring and find its index in a larger String.

Math Class and Output

Learn on this site
Math Class Static Methods

Math methods are static — called on the class, not an object. Key methods: Math.abs(x), Math.pow(base,exp), Math.sqrt(x), Math.random() → [0.0, 1.0). No import needed.

Common misconception: Math.random() returns a double in [0,1) — it never returns exactly 1.0. Scale by multiplying and casting: (int)(Math.random()*n) gives 0 to n-1.

How it can be tested: Use Math methods in expressions; calculate random integer in a range.

System.out.print vs println

System.out.println(x) prints x then newline. System.out.print(x) prints x without newline. String concatenation with + converts other types to String automatically.

Common misconception: Concatenating an int and String can give unexpected results: "x=" + 2+3 → "x=23" not "x=5". Use parentheses: "x=" + (2+3).

How it can be tested: Trace output of print/println sequences; predict output of concatenation expressions.

Final Variables and Constants

Declare a constant with final: final double PI = 3.14159; After initialization, a final variable cannot be reassigned. Convention: ALL_CAPS names for constants.

Common misconception: final prevents reassignment but does not make referenced objects immutable — it only prevents the variable from pointing to a new object.

How it can be tested: Identify final variables; explain why constants improve readability and safety.

Key Vocabulary
  • Math.random() — Returns random double in [0.0, 1.0); multiply and cast to get integer range
  • Math.pow(a,b) — Returns a^b as a double; e.g., Math.pow(2,10) → 1024.0
  • Math.sqrt(x) — Returns square root of x as a double
  • System.out.println — Prints the argument then moves to next line
  • final — Modifier that prevents reassignment of a variable after initialization
Objectives
  • Use Math class static methods in arithmetic expressions.
  • Trace output of print and println statements including concatenation.
  • Declare and use final (constant) variables.
How this can be tested
  • MCQ: predict output of a println with mixed concatenation.
  • FRQ: use Math.random() to generate a number in a specific range.
  • MCQ: identify which statement correctly declares a constant.
Practice before moving on
  • Practice generating random integers in different ranges.
  • Trace output of multiple print/println calls.

Using Existing Classes and APIs

Learn on this site
AP® CS A Quick Reference

The AP® exam provides a Quick Reference listing constructors and methods for String, Integer, Double, Math, List, and ArrayList. You must be able to read and apply method signatures from this reference.

Common misconception: You do not need to memorize exact method signatures — the exam provides them. But you must understand the parameter types and return types.

How it can be tested: Read a method signature from the Quick Reference and use it correctly in code.

Wrapper Classes: Integer and Double

Integer and Double wrap primitives as objects. Key methods: Integer.MAX_VALUE, Integer.MIN_VALUE, Integer.parseInt(s), Double.parseDouble(s). Autoboxing converts int ↔ Integer automatically.

Common misconception: Integer.MAX_VALUE is the largest int (2,147,483,647) — exceeding it causes overflow, silently wrapping to negative. AP® MCQ sometimes tests this.

How it can be tested: Use Integer and Double static methods; explain autoboxing.

Reading Preconditions and Postconditions

AP® FRQ often specifies preconditions (assumptions you can make about input) and postconditions (what the method must guarantee). You must write code that satisfies postconditions assuming preconditions hold.

Common misconception: Do NOT add extra validation code unless the FRQ requires it — if a precondition guarantees input is valid, test for it anyway wastes time and can confuse graders.

How it can be tested: Write a method that meets a given postcondition; use preconditions to simplify code.

Key Vocabulary
  • API — Application Programming Interface; the set of constructors and methods a class provides
  • Integer.parseInt(s) — Converts String s to an int; throws NumberFormatException if s is not numeric
  • autoboxing — Automatic conversion between primitive int and Integer object by Java compiler
  • precondition — An assumption guaranteed true before a method executes; stated in documentation
  • postcondition — A guarantee about what a method produces or changes; defines correct behavior
Objectives
  • Read and apply method signatures from the AP® CS A Quick Reference.
  • Use Integer.parseInt and Double.parseDouble to convert Strings.
  • Write methods whose behavior satisfies stated pre- and postconditions.
How this can be tested
  • FRQ: write a method body that satisfies a given postcondition.
  • MCQ: trace code that uses a method given only its signature.
  • MCQ: identify the effect of Integer.MAX_VALUE overflow.
Practice before moving on
  • Read all entries in the AP® CS A Quick Reference and practice calling each method.
  • Write small methods with precondition/postcondition comments.
