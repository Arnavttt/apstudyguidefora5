Boolean Expressions and Comparisons

Learn on this site
Relational Operators

Java relational operators: == != < > <= >= produce boolean values. For primitives, == compares values. For objects (Strings), == compares references; use .equals() for content.

Common misconception: Using == to compare Strings tests whether they are the exact same object, not whether they have the same characters.

How it can be tested: Evaluate relational expressions; identify correct operator for primitives vs objects.

Logical Operators

Logical operators: && (AND), || (OR), ! (NOT). Short-circuit evaluation: A && B — if A is false, B is not evaluated. A || B — if A is true, B is not evaluated.

Common misconception: Short-circuit evaluation prevents NullPointerExceptions: checking (obj != null && obj.method()) is safe; reversing the order would crash if obj is null.

How it can be tested: Evaluate compound boolean expressions; explain short-circuit evaluation.

Operator Precedence

! has higher precedence than && which has higher precedence than ||. Use parentheses to clarify complex expressions. De Morgan's Laws: !(A && B) = !A || !B; !(A || B) = !A && !B.

Common misconception: Forgetting precedence in compound conditions is a common source of logic errors. Always use parentheses in complex boolean expressions on the AP® exam.

How it can be tested: Apply De Morgan's Laws to rewrite conditions; evaluate precedence in compound expressions.

Key Vocabulary
  • boolean — Primitive type holding true or false; result of relational and logical expressions
  • && — Logical AND: true only if both operands are true; short-circuits on false left
  • || — Logical OR: true if at least one operand is true; short-circuits on true left
  • ! — Logical NOT: flips true to false and vice versa; highest logical precedence
  • De Morgan's Laws — !(A&&B) = !A||!B; !(A||B) = !A&&!B; useful for simplifying conditions
Objectives
  • Evaluate relational expressions for both primitives and objects.
  • Apply && and || with short-circuit evaluation.
  • Use De Morgan's Laws to rewrite boolean conditions.
How this can be tested
  • MCQ: evaluate a compound boolean expression step-by-step.
  • FRQ: write a condition that combines multiple constraints.
  • MCQ: identify the equivalent of a negated compound condition using De Morgan's Laws.
Practice before moving on
  • Evaluate truth tables for &&, ||, ! with different inputs.
  • Rewrite negated conditions using De Morgan's Laws.

If and If-Else Statements

Learn on this site
If, Else-If, Else

if (cond) {body} else if (cond2) {body2} else {body3}. Braces recommended even for single statements. Only one branch executes per run. Else catches everything not matched above.

Common misconception: Omitting braces in if-else chains makes indentation-based reasoning dangerous — Java ignores indentation.

How it can be tested: Trace which branch executes for given inputs; write conditions for range-based classification.

Nested If Statements

An if statement inside another if. Useful when a second decision depends on the first. May be rewritten as compound condition: if (A && B) {...} instead of nested ifs.

Common misconception: The "dangling else" problem: else attaches to the nearest unmatched if. Always use braces to prevent this ambiguity.

How it can be tested: Trace nested if-else behavior; rewrite as flat if-else chain when possible.

Common If-Else Patterns

Testing ranges: if (x < 0) ... else if (x == 0) ... else ... Checking divisibility: if (n%2 == 0). Assigning based on condition can use if-else or the ternary operator: type var = (cond) ? a : b;

Common misconception: Range tests must be mutually exclusive and exhaustive. Overlapping conditions cause logic bugs.

How it can be tested: Write range-based if-else for grading or classification; use modulo for divisibility.

Key Vocabulary
  • if — Executes a block only when the condition is true
  • else if — Alternative condition checked only when previous conditions were false
  • else — Executes when all preceding conditions were false; no condition needed
  • dangling else — Ambiguity when else matches nearest unmatched if — braces eliminate this
  • ternary operator — type var = (cond) ? valueIfTrue : valueIfFalse; compact if-else for assignment
Objectives
  • Write if/else-if/else chains for range-based classification.
  • Trace nested if-else to determine which block executes.
  • Identify and fix the dangling else problem.
How this can be tested
  • FRQ: write a method that returns a String based on a numeric input range.
  • MCQ: trace nested if-else to find the value printed.
  • MCQ: identify the output of an if-else chain with a specific input.
Practice before moving on
  • Write a grade calculator using if-else-if for score ranges.
  • Trace a nested if-else with truth tables for inputs.

For and While Loops

Learn on this site
For Loops

for (init; condition; update) {body}. Executes init once, checks condition, runs body, runs update, repeats. Loop variable scope limited to loop unless declared outside. Common: for (int i=0; i

Common misconception: Off-by-one errors: i

How it can be tested: Trace a for loop for given start/end values; count iterations; identify off-by-one errors.

While Loops

while (condition) {body}. Checks condition before each iteration. If condition is initially false, body never executes. Use when the number of iterations is not known in advance.

Common misconception: An infinite loop results if the loop variable is never updated or the condition can never become false. Always ensure the loop makes progress toward termination.

How it can be tested: Write while loops for input validation and search patterns; trace execution.

do-while and Loop Comparison

do {body} while (condition); executes body at least once. Less common in AP® CS A. Choosing for vs while: for when count is known; while when condition-driven; do-while for at-least-once scenarios.

Common misconception: AP® CS A rarely tests do-while directly, but knowing it prevents confusion when reading code that contains one.

How it can be tested: Choose the correct loop type for a given scenario; convert between for and while loops.

Key Vocabulary
  • for loop — Loop with init, condition, update in header; best when iteration count is known
  • while loop — Loop that repeats while condition is true; best for condition-driven iteration
  • do-while — Loop that always executes body at least once before checking condition
  • off-by-one error — Using < vs <= incorrectly, causing one too many or too few iterations
  • infinite loop — Loop whose condition never becomes false; program hangs without terminating
Objectives
  • Write for loops that iterate n times with correct boundaries.
  • Write while loops for condition-driven scenarios.
  • Identify and fix off-by-one errors in loop boundaries.
How this can be tested
  • FRQ: write a loop to traverse an array or compute a sum.
  • MCQ: count the number of iterations of a given loop.
  • MCQ: trace a while loop to find the output.
Practice before moving on
  • Count iterations and trace values for various loop bounds.
  • Convert a for loop to an equivalent while loop.

Nested Loops and String Traversal

Learn on this site
Nested Loops

A loop inside another loop. Outer loop runs n times; inner loop runs m times per outer iteration → total n×m iterations. Used for 2D patterns, comparing all pairs, and matrix traversal.

Common misconception: Forgetting that the inner loop counter resets each time the outer loop runs is a common tracing error.

How it can be tested: Trace nested loops to predict output; count total iterations; write nested loops for rectangular patterns.

String Traversal with Loops

Use a for loop with i from 0 to str.length()-1: char c = str.charAt(i) or str.substring(i,i+1). Process each character individually — search, count, or transform.

Common misconception: charAt returns a char (primitive); substring(i,i+1) returns a String of length 1. Comparing chars: use == for char; use .equals() for String.

How it can be tested: Write a loop to count occurrences of a character in a String; trace character extraction.

Loop Invariants and Tracing

A loop invariant is a condition that holds before and after each iteration. Useful for reasoning about correctness. Track accumulator variables carefully when tracing.

Common misconception: When tracing, track ALL variables simultaneously through each iteration — focusing only on the loop counter while ignoring the accumulator causes incorrect traces.

How it can be tested: Trace a loop with accumulator to find the final value; identify what a loop computes.

Key Vocabulary
  • nested loop — A loop inside another loop; inner loop completes all iterations for each outer iteration
  • charAt(i) — Returns the char at index i of a String; char is a primitive
  • accumulator — Variable that collects a running total or result across loop iterations
  • loop invariant — A condition true before and after every iteration; helps verify correctness
  • char — Primitive type holding a single character; compared with == not .equals()
Objectives
  • Trace nested loops to count iterations and predict output.
  • Write a loop to traverse each character of a String.
  • Identify what an accumulator computes after a loop completes.
How this can be tested
  • FRQ: write a method that counts or extracts characters from a String.
  • MCQ: trace nested loops to find output pattern.
  • MCQ: identify the number of total iterations in a nested loop.
Practice before moving on
  • Trace a nested loop that prints a triangle pattern.
  • Write a method that reverses a String using a loop.
