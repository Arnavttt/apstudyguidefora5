Designing a Class

Learn on this site
Class Structure

A class defines a type with fields (data) and methods (behavior). Fields are declared at class scope; local variables exist only within methods. Fields have default values (0, false, null) if not initialized.

Common misconception: Instance variables belong to the object; local variables belong to the method. They have separate scopes and lifetimes.

How it can be tested: Identify which variables are instance vs local; design a class for a given real-world entity.

Writing a Basic Class

class ClassName { fields; constructors; methods; }. By convention class names are UpperCamelCase. Separate the class definition from its use (test in another class).

Common misconception: AP® FRQ often gives you a partial class and asks you to add a method. Read the existing fields and methods first to understand the design before writing.

How it can be tested: Add a method to a given class skeleton; identify the class's fields and their types.

Object State and Behavior

State = values of instance variables at a moment in time. Behavior = what methods do. An object's state can change through mutator methods; behavior is defined by the class.

Common misconception: Two objects of the same class are independent — changing one does not change the other unless explicitly connected.

How it can be tested: Describe the state of an object after a series of method calls; trace state changes.

Key Vocabulary
  • instance variable — Variable declared at class scope; each object has its own copy; persists for the object's lifetime
  • local variable — Variable declared inside a method; exists only during that method's execution
  • class — Blueprint defining fields and methods; instantiated to create objects
  • state — Values of an object's instance variables at a point in time
  • behavior — What an object can do; defined by its methods
Objectives
  • Design a class with appropriate fields for a given problem.
  • Distinguish instance variables from local variables by scope.
  • Describe an object's state after a sequence of method calls.
How this can be tested
  • FRQ: write a class skeleton with fields and method signatures for a described problem.
  • MCQ: trace state changes of an object through method calls.
  • FRQ: add a method to an existing class using its fields.
Practice before moving on
  • Design a Student class with fields for name, grade, and GPA.
  • Trace object state through a series of method calls.

Constructors

Learn on this site
Constructor Syntax

A constructor has the same name as the class and no return type. Called with new: ClassName obj = new ClassName(args). Used to initialize fields.

Common misconception: A constructor is NOT a method — it has no return type, not even void. Writing void before a constructor name creates an ordinary method instead.

How it can be tested: Write a constructor that initializes all fields; identify constructor vs method.

Constructor Overloading

Multiple constructors can exist with different parameter lists. The default (no-arg) constructor is provided only if NO constructors are defined; once you write one, the default is gone.

Common misconception: If you define a parameterized constructor without a no-arg constructor, code using new ClassName() will fail to compile.

How it can be tested: Write overloaded constructors; identify which constructor matches a given new call.

this Keyword

this refers to the current object. Used to resolve shadowing: this.name = name; Also used to call another constructor: this(args); must be the first statement.

Common misconception: this() can only be used as the first statement in a constructor. It delegates to another overloaded constructor of the same class.

How it can be tested: Use this to disambiguate field names; use this() for constructor chaining.

Key Vocabulary
  • constructor — Special method with class name and no return type; initializes object fields
  • overloading — Defining multiple methods/constructors with the same name but different parameter lists
  • default constructor — No-arg constructor provided automatically only if no constructors are defined
  • this — Reference to the current object; resolves field-parameter shadowing
  • constructor chaining — Calling this(args) from a constructor to delegate to another constructor
Objectives
  • Write constructors that initialize all instance variables.
  • Explain why the default no-arg constructor is lost when you define any constructor.
  • Use this to assign shadowed fields correctly.
How this can be tested
  • FRQ: write two constructors — a full parameterized one and a no-arg default.
  • MCQ: identify which constructor is called by a given new expression.
  • MCQ: identify a constructor error (missing void? wrong name?).
Practice before moving on
  • Write a BankAccount class with two constructors: one sets balance to 0; one accepts initial balance.
  • Practice constructor chaining with this().

Encapsulation: Accessors and Mutators

Learn on this site
Private Fields

Declaring fields private prevents direct access from outside the class. This is encapsulation — hiding internal state. Public fields can be changed by anyone; private fields can only be changed through methods you control.

Common misconception: On the AP® exam, all instance variables should be declared private. Making them public is a design flaw (loses encapsulation).

How it can be tested: Identify which fields should be private; explain why encapsulation prevents invalid state.

Accessor (Getter) Methods

An accessor returns the value of a private field without allowing modification. Convention: getName(), getBalance(), isComplete(). Return type matches the field type.

Common misconception: AP® FRQ often requires you to write a getter. The method must have the correct return type and use return, not print.

How it can be tested: Write accessor methods for all private fields; return the correct type.

Mutator (Setter) Methods

A mutator changes a private field's value, often with validation. Convention: setName(String name), deposit(double amount). Return type is usually void.

Common misconception: A setter without validation is little better than a public field. AP® FRQ sometimes requires adding validation (e.g., if amount > 0 before depositing).

How it can be tested: Write mutators with simple validation; explain how they protect object invariants.

Key Vocabulary
  • encapsulation — Hiding fields as private and exposing controlled access through methods
  • accessor — Method that returns the value of a private field; getter; return type matches field
  • mutator — Method that changes a private field, often with validation; setter; usually void
  • private — Access modifier restricting field/method to within the class only
  • public — Access modifier allowing access from any other class
Objectives
  • Declare all instance variables private.
  • Write accessor (getter) methods that return field values.
  • Write mutator (setter) methods that update fields with validation.
How this can be tested
  • FRQ: write getters and setters for a given class skeleton.
  • MCQ: identify an encapsulation violation.
  • FRQ: add a mutator that validates input before updating a field.
Practice before moving on
  • Add getters and setters to a BankAccount class.
  • Explain why a public field can lead to invalid state.

Static Members and Method Design

Learn on this site
Static vs Instance Members

static fields and methods belong to the class, not to any object. Called with ClassName.method(). Instance methods need an object. Math.sqrt() and Integer.MAX_VALUE are static. Use static for shared data or utilities.

Common misconception: Calling a static method on an object (obj.staticMethod()) compiles but is misleading — the call ignores the object. Call static methods on the class name.

How it can be tested: Distinguish static from instance; call static methods correctly using the class name.

Writing Complete Methods

A method has: access modifier, return type (or void), name, parameters, and body. Return type must match what is returned. void methods may contain return; with no value to exit early.

Common misconception: Forgetting the return statement in a non-void method causes a compile error. Every path through the method must end in a return.

How it can be tested: Write complete method implementations with correct return types; ensure all paths return a value.

toString and equals Methods

toString() returns a String representation of the object: @Override public String toString(). equals(Object other) checks content equality. Both inherited from Object but should be overridden.

Common misconception: Without overriding toString, printing an object shows a cryptic address like Dog@7a81197d — not useful for debugging or output.

How it can be tested: Override toString to produce a readable representation; use it in output statements.

Key Vocabulary
  • static — Modifier making a field or method belong to the class, not a specific object
  • instance method — Method requiring an object to be called; accesses instance variables
  • static method — Method called on the class name; cannot access instance variables
  • toString() — Returns a String representation of the object; called automatically in concatenation/println
  • void — Return type for methods that produce no value; may use return; to exit early
Objectives
  • Write static utility methods and call them on the class name.
  • Write complete methods with all paths returning the correct type.
  • Override toString to produce meaningful output for an object.
How this can be tested
  • FRQ: write an instance method that uses fields and returns a computed value.
  • MCQ: identify whether a method should be static or instance.
  • FRQ: write a toString method for a given class.
Practice before moving on
  • Add a static counter field to a class counting how many objects have been created.
  • Write toString for a Point class: "Point(x, y)".
