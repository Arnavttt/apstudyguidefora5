Algorithms and Pseudocode

Learn on this site
What is an Algorithm?

An algorithm is a finite, unambiguous sequence of steps that solves a problem. Properties: finite (terminates), unambiguous (each step is clear), correct (produces right output for all valid inputs).

Common misconception: An algorithm must terminate — an infinite loop is not an algorithm, even if it produces some output.

How it can be tested: Identify whether a described process is a valid algorithm; explain why a non-terminating process is not.

AP® CSP Pseudocode

AP® CSP uses a specific pseudocode notation: DISPLAY(), INPUT(), assignment with ←, IF/ELSE/END IF, REPEAT n TIMES / UNTIL / END REPEAT, lists, and procedures. This pseudocode is used on the exam.

Common misconception: AP® CSP pseudocode uses ← for assignment (not = or :=). Confusing notation with Python or Java on the exam causes errors.

How it can be tested: Read and trace AP® CSP pseudocode; convert pseudocode to plain English description.

Expressing Algorithms

The same algorithm can be expressed as pseudocode, flowchart, natural language, or code. The expression does not change the algorithm itself. Flowcharts: rectangles = actions, diamonds = decisions, ovals = start/end.

Common misconception: Flowcharts and pseudocode describe the same algorithm — the representation does not affect what the algorithm does.

How it can be tested: Convert between pseudocode, flowchart, and English description of the same algorithm.

Key Vocabulary
  • algorithm — Finite, unambiguous sequence of steps that solves a problem and terminates
  • pseudocode — Human-readable code-like notation describing an algorithm without language-specific syntax
  • flowchart — Visual representation of an algorithm using shapes for steps and decisions
  • ← (assignment) — AP® CSP notation for storing a value: x ← 5 sets x to 5
  • REPEAT UNTIL — AP® CSP loop that repeats until a condition becomes true
Objectives
  • Define algorithm and identify its required properties.
  • Trace AP® CSP pseudocode step-by-step to find variable values and output.
  • Convert between pseudocode, flowchart, and English description.
How this can be tested
  • MCQ: trace AP® CSP pseudocode to determine output.
  • MCQ: identify whether a described process is a valid algorithm.
  • Written response: describe in English what a pseudocode algorithm does.
Practice before moving on
  • Trace AP® CSP pseudocode with loops and conditionals for given input.
  • Draw a flowchart for a simple decision algorithm.

Lists and Procedures in AP® CSP

Learn on this site
Lists in AP® CSP

Lists store ordered collections. Notation: list[i] accesses index i (1-indexed in AP® CSP pseudocode, unlike Java's 0-indexed!). Append: INSERT(list, len+1, val). Remove: REMOVE(list, i). LENGTH(list) returns size.

Common misconception: AP® CSP pseudocode lists are 1-indexed: list[1] is the first element. This is different from Java/Python where indices start at 0.

How it can be tested: Trace list operations in AP® CSP pseudocode; predict list contents after insertions and removals.

Procedures (Functions) in AP® CSP

PROCEDURE name(params) { body } defines a procedure. Call by name with arguments. RETURN(value) sends a result back. Procedures with no return produce a side effect (display, modify list).

Common misconception: A procedure that does not RETURN a value cannot be used in an expression — it can only be called as a statement.

How it can be tested: Write a PROCEDURE in AP® CSP pseudocode; call procedures and use their return values.

Iteration and Traversal

FOR EACH item IN list iterates all elements. Access index with FOR i ← 1 TO LENGTH(list) in pseudocode. Use iteration to compute sum, find max, search, or filter a list.

Common misconception: When using FOR EACH, you cannot modify the list being traversed or track the index. Use index-based loop when you need the position.

How it can be tested: Write pseudocode to traverse a list and compute an aggregate or filter elements.

Key Vocabulary
  • list (AP® CSP) — Ordered collection of values; indices start at 1 in AP® CSP pseudocode
  • LENGTH(list) — Returns the number of elements in a list
  • INSERT(list, i, val) — Inserts val at index i, shifting later elements
  • PROCEDURE — Named block of code in AP® CSP pseudocode; may accept parameters and RETURN a value
  • RETURN — Exits a PROCEDURE and optionally passes a value back to the caller
Objectives
  • Trace list operations including access, insert, remove, and length.
  • Write a PROCEDURE in AP® CSP pseudocode with parameters and return.
  • Use FOR EACH to traverse a list and compute a result.
How this can be tested
  • MCQ: trace list operations in pseudocode to find the final list contents.
  • MCQ: identify what a procedure returns for given input.
  • Written response: write a procedure that filters a list.
Practice before moving on
  • Trace a FOR EACH loop that sums a list.
  • Write a procedure that returns the minimum value of a list.

Boolean Logic and Conditionals

Learn on this site
Boolean Operators in AP® CSP

AP® CSP pseudocode uses AND, OR, NOT. Truth: A AND B is true only if both are true. A OR B is true if at least one is true. NOT A flips the value. Conditional: IF(condition) THEN block ELSE block END IF.

Common misconception: AP® CSP uses AND/OR/NOT (English words), not &&/||/! (Java) or and/or/not (Python). Know all three styles.

How it can be tested: Evaluate AP® CSP Boolean expressions; trace IF statements in pseudocode.

Nested Conditionals

IF inside another IF. Can often be rewritten as a compound AND condition. Nested conditionals are harder to read — prefer flat structure with compound conditions when possible.

Common misconception: Deeply nested conditionals are hard to debug — consider refactoring to a procedure or using compound boolean expressions.

How it can be tested: Trace nested conditionals; rewrite as compound conditions.

Selection vs Iteration

Selection (IF/ELSE) chooses between paths. Iteration (REPEAT/FOR) repeats a block. Many algorithms combine both. Choosing the right structure for a problem is a key skill.

Common misconception: Not all repetition requires a counter — REPEAT UNTIL is better when you don't know how many iterations are needed.

How it can be tested: Identify the appropriate control structure for a described algorithm.

Key Vocabulary
  • AND — Boolean operator: true only if both operands are true
  • OR — Boolean operator: true if at least one operand is true
  • NOT — Boolean operator: reverses the truth value
  • selection — Control structure that chooses one of multiple paths (IF/ELSE)
  • iteration — Control structure that repeats a block (REPEAT/FOR EACH/REPEAT UNTIL)
Objectives
  • Evaluate AP® CSP Boolean expressions with AND, OR, NOT.
  • Trace nested conditionals to determine which path executes.
  • Choose the correct control structure (selection vs iteration) for a problem.
How this can be tested
  • MCQ: evaluate a compound Boolean expression.
  • MCQ: trace an IF-ELSE pseudocode block for given input.
  • Written response: describe when REPEAT UNTIL is better than REPEAT n TIMES.
Practice before moving on
  • Build a truth table for A AND NOT B OR C.
  • Rewrite a nested IF as a single IF with compound condition.

Algorithm Efficiency and Undecidable Problems

Learn on this site
Algorithm Efficiency

Efficiency measured by how runtime grows with input size. Linear search: O(n) — check each element. Binary search: O(log n) — halve search space each step (requires sorted data). Sorting: O(n log n) for efficient sorts.

Common misconception: O(n) and O(log n) are big-O notation — they describe growth rate, not exact time. An O(n) algorithm on a slow computer can be slower than O(n²) on a fast computer for small n.

How it can be tested: Compare linear and binary search; explain why binary search requires sorted data.

Heuristics and Approximation

Some problems are too complex to solve optimally in reasonable time. Heuristics: algorithms that find a "good enough" solution quickly. Example: traveling salesman problem uses heuristics because exact solution is exponential.

Common misconception: A heuristic does NOT guarantee the optimal answer — it finds a reasonable one quickly. The trade-off is quality for speed.

How it can be tested: Explain what a heuristic is and when it is used; identify that a heuristic may not find the optimal solution.

Undecidable Problems

Some problems cannot be solved by any algorithm for all inputs. The Halting Problem: no algorithm can decide whether an arbitrary program will halt for all inputs. Undecidable ≠ unsolvable for specific cases.

Common misconception: Undecidable does not mean the problem is hard — it means no algorithm exists that always gives the correct answer for all possible inputs.

How it can be tested: Explain what an undecidable problem is; describe the Halting Problem in plain language.

Key Vocabulary
  • linear search — Check each element sequentially; O(n); works on any list
  • binary search — Halve search space each step; O(log n); requires sorted list
  • heuristic — Algorithm that finds a good-enough solution quickly; may not be optimal
  • undecidable problem — Problem for which no algorithm can always give the correct answer for all inputs
  • Halting Problem — Classic undecidable problem: no algorithm can decide if an arbitrary program halts
Objectives
  • Compare linear and binary search in terms of efficiency and requirements.
  • Explain what a heuristic is and when it is appropriate.
  • Describe the Halting Problem and what it means for a problem to be undecidable.
How this can be tested
  • MCQ: identify the more efficient algorithm for a given scenario.
  • Written response: explain why binary search requires a sorted list.
  • MCQ: identify a problem as decidable, hard-but-solvable, or undecidable.
Practice before moving on
  • Trace binary search on a sorted list to find a target.
  • Describe a real-world heuristic (GPS routing, spell check).
