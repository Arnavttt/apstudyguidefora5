Computer Hardware

Learn on this site
Components of a Computer

CPU: Central Processing Unit — executes instructions. RAM: Random Access Memory — fast, temporary storage. Storage (HDD/SSD): permanent storage. I/O devices: keyboard, display, network interface.

Common misconception: RAM is volatile — contents are lost when power is off. Storage (HDD/SSD) is non-volatile — data persists. Confusing them is a common misconception.

How it can be tested: Identify the role of CPU, RAM, and storage; explain the volatile vs non-volatile distinction.

CPU and Processing

CPU executes the fetch-decode-execute cycle: fetch instruction from memory, decode it, execute. Clock speed (GHz) and number of cores affect performance. More cores allow parallel tasks.

Common misconception: More cores do not always mean faster performance — software must be written to use multiple cores (parallel programming).

How it can be tested: Explain the fetch-decode-execute cycle; describe how multiple cores improve performance.

Hardware Abstraction

Operating system (OS) abstracts hardware from applications. Applications see a file system, not raw disk sectors. Drivers translate between OS and specific hardware. This layering is a form of abstraction.

Common misconception: The OS is not optional — without it, every application would need to directly manage hardware, which is impractical.

How it can be tested: Explain the role of the OS in abstracting hardware; describe the driver concept.

Key Vocabulary
  • CPU — Executes instructions; the "brain" of the computer; measured in GHz and cores
  • RAM — Volatile fast-access memory holding currently running programs; lost on power off
  • storage (SSD/HDD) — Non-volatile permanent storage; slower than RAM but retains data without power
  • operating system — Software managing hardware resources and providing services to applications
  • driver — Software translating OS requests to specific hardware commands
Objectives
  • Identify CPU, RAM, and storage and explain their roles.
  • Explain the fetch-decode-execute cycle.
  • Describe how the OS abstracts hardware for applications.
How this can be tested
  • MCQ: identify which component is volatile or non-volatile.
  • MCQ: describe the role of the operating system.
  • Written response: explain how multiple cores improve performance.
Practice before moving on
  • Diagram the relationship between CPU, RAM, storage, and OS.
  • Explain why an app does not need to know the specific hard drive brand.

The Internet and Protocols

Learn on this site
How the Internet Works

The internet is a global network of networks. Data is broken into packets sent independently and reassembled at destination. Routers direct packets using the best available path. No central authority controls routing.

Common misconception: The internet is decentralized — packets from the same message can travel different routes and arrive out of order. TCP/IP handles reassembly.

How it can be tested: Explain packet switching and why it makes the internet fault-tolerant.

Protocols: TCP/IP and HTTP

IP (Internet Protocol): addresses and routes packets. TCP (Transmission Control Protocol): ensures delivery, order, and error checking. HTTP/HTTPS: application protocol for web pages. DNS: translates domain names to IP addresses.

Common misconception: HTTP is insecure — data is plain text. HTTPS adds TLS encryption. Always use HTTPS for sensitive data.

How it can be tested: Explain the role of IP, TCP, DNS, and HTTP/HTTPS in web communication.

Bandwidth and Latency

Bandwidth: maximum data per unit time (Mbps). Latency: time for a packet to travel from source to destination (ms). High bandwidth ≠ low latency. A satellite link has high bandwidth but high latency.

Common misconception: Downloading a large file benefits from high bandwidth. Online gaming requires low latency. These are different bottlenecks.

How it can be tested: Distinguish bandwidth from latency; identify which matters more for a given application.

Key Vocabulary
  • packet — Fixed-size chunk of data with header (destination, source, sequence) and payload
  • router — Network device that forwards packets toward their destination
  • IP address — Numeric label (e.g., 192.168.1.1) identifying a device on a network
  • DNS — Domain Name System: translates domain names (www.example.com) to IP addresses
  • bandwidth — Maximum data transfer rate (Mbps or Gbps); determines download speed
Objectives
  • Explain how packets travel from source to destination on the internet.
  • Describe the roles of IP, TCP, DNS, and HTTP.
  • Distinguish bandwidth from latency.
How this can be tested
  • MCQ: identify the protocol responsible for a specific internet function.
  • MCQ: distinguish bandwidth from latency in a given scenario.
  • Written response: explain how packet switching supports fault tolerance.
Practice before moving on
  • Trace the path of a web request from typing a URL to seeing the page.
  • Explain why HTTPS should be used for banking websites.

Cybersecurity

Learn on this site
Common Threats

Phishing: fake sites/emails steal credentials. Malware: software designed to damage or steal (virus, ransomware, spyware). Man-in-the-middle attack: attacker intercepts communication. DDoS: flood a server with requests to make it unavailable.

Common misconception: Phishing is more effective than technical hacking — social engineering exploits human trust, not software vulnerabilities.

How it can be tested: Identify types of cybersecurity threats; explain how each attack works.

Defenses

Encryption protects data in transit and at rest. Two-factor authentication (2FA) adds a second factor (code, biometric). Firewalls filter network traffic. Regular updates patch vulnerabilities. Strong, unique passwords for each account.

Common misconception: A strong password cannot protect against phishing — if you enter it on a fake site, the attacker has it. 2FA limits damage even when credentials are stolen.

How it can be tested: Describe encryption, 2FA, and firewalls as defenses; explain why 2FA improves security over passwords alone.

Encryption

Symmetric encryption: same key to encrypt and decrypt (fast). Asymmetric (public-key): different keys — public key encrypts, private key decrypts (RSA). HTTPS uses asymmetric to exchange symmetric key, then symmetric for data.

Common misconception: HTTPS is not "unbreakable" — it protects data in transit but cannot protect against a compromised server or a phished password.

How it can be tested: Explain symmetric vs asymmetric encryption; describe the role of HTTPS in securing web traffic.

Key Vocabulary
  • phishing — Fraudulent emails or sites that trick users into revealing credentials
  • malware — Malicious software: viruses, ransomware, spyware, trojans
  • encryption — Converting data to an unreadable form; only decryptable with the correct key
  • symmetric encryption — Uses the same key for encryption and decryption; fast
  • asymmetric encryption — Public key encrypts; private key decrypts; used in HTTPS key exchange
Objectives
  • Identify cybersecurity threats and explain how they work.
  • Describe encryption, 2FA, and firewall defenses.
  • Explain symmetric vs asymmetric encryption.
How this can be tested
  • MCQ: identify a described attack type (phishing, DDoS, MITM).
  • Written response: explain how 2FA improves security over passwords alone.
  • MCQ: identify whether symmetric or asymmetric encryption is used.
Practice before moving on
  • Evaluate the security of two passwords using strength criteria.
  • Explain what happens to data in a man-in-the-middle attack on HTTP.

Fault Tolerance and Parallel Computing

Learn on this site
Fault Tolerance

Fault tolerance: ability to continue operating despite component failures. The internet achieves this through redundancy — multiple paths exist so packets route around failures. Redundant servers store the same data.

Common misconception: Fault tolerance does not mean the system never fails — it means failures do not cause complete system shutdown.

How it can be tested: Explain how redundancy achieves fault tolerance; give examples of fault-tolerant systems.

Parallel Computing

Parallel: multiple tasks run simultaneously. Sequential: one task at a time. Parallelism requires tasks to be independent. Speedup limited by the sequential portion (Amdahl's Law). Benefits: faster processing of large tasks (weather modeling, rendering).

Common misconception: Not all problems can be parallelized — if step 2 requires output from step 1, they must run sequentially. Amdahl's Law shows the limit.

How it can be tested: Identify which tasks can run in parallel and which must be sequential; explain Amdahl's Law.

Cloud Computing

Cloud: computing resources (servers, storage, databases) provided over the internet. Benefits: scalability, reduced hardware cost, global access. Risks: data privacy, vendor lock-in, single point of failure if vendor has outage.

Common misconception: Cloud computing does not eliminate security concerns — it shifts some responsibility to the cloud provider but the user still must manage access and data.

How it can be tested: Describe cloud computing benefits and risks; distinguish IaaS, PaaS, and SaaS.

Key Vocabulary
  • redundancy — Duplicating components so failure of one does not stop the system
  • fault tolerance — System's ability to continue operation despite individual component failures
  • parallel computing — Multiple computations happening simultaneously on independent tasks
  • Amdahl's Law — Speedup limited by sequential portion: if 50% must be sequential, max speedup is 2×
  • cloud computing — Delivering computing resources over the internet on demand
Objectives
  • Explain how redundancy achieves fault tolerance in the internet.
  • Identify tasks that can be parallelized and those that cannot.
  • Describe cloud computing benefits and associated risks.
How this can be tested
  • MCQ: identify a fault-tolerant design feature.
  • MCQ: determine whether two tasks can run in parallel given dependencies.
  • Written response: describe a trade-off of using cloud storage.
Practice before moving on
  • Identify which steps in a recipe can be done in parallel vs sequentially.
  • Explain how cloud storage like Google Drive achieves fault tolerance.
